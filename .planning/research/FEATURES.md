# Feature Research

**Domain:** 教育管理系统 (Education Management System)
**Researched:** 2026-03-30
**Confidence:** HIGH

## Feature Landscape

### Table Stakes (标配)

用户默认期望的功能。缺少这些功能 = 产品不完整，无法使用。

| 功能 | 为什么必须 | 复杂度 | 备注 |
|------|-----------|--------|------|
| 用户认证与登录 | 系统安全基础，所有用户必须通过身份验证才能使用 | LOW | JWT 双令牌机制已实现 |
| 学生信息管理 (CRUD) | 学籍管理是教务系统核心，存储学生基本信息、联系方式、学籍状态 | LOW | 项目待开发模块 |
| 教师信息管理 | 教师是教学主体，需管理教师基本信息、科目、资质 | LOW | 项目待开发模块 |
| 班级管理 | 班级是教学组织基本单位，需支持创建、编排、班主任分配 | LOW | 项目待开发模块 |
| 课程管理 | 课程是教学核心实体，需管理课程信息、课时、学分 | LOW | 项目待开发模块 |
| 成绩管理 | 成绩是教学评估核心，需支持成绩录入、查看、统计 | MEDIUM | 项目待开发模块，需考虑权限控制 |
| 考勤管理 | 记录学生出勤情况，与成绩、评优关联 | MEDIUM | 项目待开发模块 (leaves) |
| 权限管理 (RBAC) | 不同角色(管理员、教师、学生)需要不同访问权限 | MEDIUM | 项目待开发模块 (roles/permissions) |
| 数据字典 | 系统配置数据(如年级、学期状态)需要统一管理 | LOW | 项目待开发模块 (dicts) |
| 菜单管理 | 动态菜单支撑权限控制，菜单结构需可配置 | LOW | 项目待开发模块 (menus) |

### Differentiators (差异化)

创造竞争优势的功能。非必须，但能显著提升产品竞争力和用户粘性。

| 功能 | 价值主张 | 复杂度 | 备注 |
|------|---------|--------|------|
| 智能排课 | 自动处理排课冲突(教师时间、教室占用)，显著降低教务工作量 | HIGH | 2026年新高考改革驱动需求增长，项目待开发 |
| 数据分析与报表 | 多维度学情分析(成绩趋势、班级对比)，辅助教学决策 | MEDIUM | 可集成 ECharts (前端已有示例) |
| 家校互通平台 | 家长实时接收出勤、成绩、通知，提升服务满意度 | MEDIUM | 培训机构刚需，公立学校也在普及 |
| 移动端体验 (小程序/H5) | 家长、教师移动办公必备，支持请假、查成绩 | MEDIUM | 项目计划外，需评估范围 |
| 在线考试与作业 | 支持在线组卷、自动批改、题库管理 | HIGH | 在线教育趋势，K12培训机构刚需 |
| 招生管理 (CRM) | 线索管理、转化漏斗、续费提醒，培训机构核心需求 | MEDIUM | 项目当前范围外 |
| 财务与收费 | 学费管理、账单生成、在线支付 | MEDIUM | 项目当前范围外 |

### Anti-Features (反功能)

看起来有用但实际会产生问题的功能。

| 功能 | 为什么被请求 | 为什么有问题 | 替代方案 |
|------|-------------|-------------|---------|
| 过度复杂的 AI 排课 | "想要最智能的排课" | 算法复杂度指数级增长，实际效果可能不如简单规则 | 先实现基础规则排课，积累后再升级 AI |
| 完全离线优先 | "网络不稳定需要离线用" | 离线同步逻辑复杂，维护成本高，2026年网络覆盖已良好 | 聚焦核心业务，暂不考虑离线 |
| 高度可定制化表单 | "每个学校流程不同" | 过度定制导致系统不稳定，维护成本高 | 提供标准流程 + 有限配置项 |
| 第三方社交登录 | "用户不想注册" | 增加安全复杂度，弱密码风险 | 坚持密码+验证码登录，确保安全 |
| 桌面客户端 | "需要离线桌面版" | 与 Web 端功能重复，开发维护成本高 | 项目已有 Dioxus 桌面端计划，可后续评估 |

## Feature Dependencies

```
用户认证
    └──需要──> 权限管理 ──需要──> 菜单管理

班级管理
    └──需要──> 学生管理 (学生需关联班级)

课程管理
    └──需要──> 排课管理 (课程是排课输入)

成绩管理
    └──需要──> 考试管理 (成绩来源于考试)
    └──需要──> 学生管理 (成绩需关联学生)
    └──需要──> 课程管理 (成绩需关联课程)

排课管理
    └──需要──> 课程管理
    └──需要──> 班级管理
    └──需要──> 教师管理
    └──需要──> 教室资源管理

考勤/请假
    └──需要──> 学生管理
    └──需要──> 班级管理
```

### Dependency Notes

- **成绩管理依赖考试管理：** 成绩必须来源于具体的考试场次，需先实现考试管理
- **排课依赖课程/班级/教师：** 排课是下游功能，需要上游基础数据完善后才能实现
- **权限管理依赖用户认证：** 权限控制建立在身份认证之上，是认证的自然延伸
- **智能排课与基础排课不冲突：** 可先实现基础手工排课，再叠加智能算法

## MVP Definition

### Launch With (v1)

最小可用产品 — 验证核心价值所需的最少功能集。

- [x] 用户认证与登录 — 系统基础，必须有
- [x] 用户管理 CRUD — 管理员账户管理，已实现
- [ ] 学生信息管理 — 核心业务，必须有
- [ ] 教师信息管理 — 核心业务，必须有
- [ ] 班级管理 — 核心业务，必须有
- [ ] 课程管理 — 核心业务，必须有
- [ ] 成绩管理 — 核心业务，必须有
- [ ] 权限管理 (RBAC) — 安全基础，必须有
- [ ] 数据字典 — 系统配置基础，必须有
- [ ] 菜单管理 — 权限展示基础，必须有

**MVP 评估：** 项目 PROJECT.md 中列出的待实现模块基本覆盖了 MVP 范围

### Add After Validation (v1.x)

核心验证完成后可添加的功能。

- [ ] 考勤管理 — 与成绩关联，可提升系统完整性
- [ ] 考试管理 — 成绩录入的前置条件
- [ ] 基础排课功能 — 手工排课，满足基本需求
- [ ] 数据分析与报表 — 教学决策支持

### Future Consideration (v2+)

建立产品市场匹配后再考虑的功能。

- [ ] 智能排课算法 — 需要真实用户反馈才能优化
- [ ] 家校互通平台 — 需要评估目标用户需求
- [ ] 在线考试与题库 — 需要较大研发投入
- [ ] 移动端小程序 — 需要评估投入产出比
- [ ] 招生管理 CRM — 培训机构特定需求
- [ ] 财务与收费模块 — 需了解具体业务场景

- **培训机构市场：** 校宝在线是领导者，功能全但复杂度高，Fayzedu 可聚焦简单易用

- **Fayzedu 差异化定位：** 轻量级、高性能、技术栈现代(Rust+Vue3)，适合技术团队自建或二次开发
